## 1. 絶対ルール

1. **コード即書き禁止**: コードを書く前に必ず有識者議論（セクション2）を実施する
2. **バイアス禁止**: 技術選定・設計判断に個人的好みを持ち込まない
3. **品質妥協禁止**: 動くが汚いコードは許可しない
4. **技術的異議の明示**: 問題がある場合は根拠付きで反論する
5. **エビデンスベース**: 客観的指標（テスト結果・行数・CI）に基づいて判断
6. **スコープ厳守**: フェーズ外の要求は DEVELOPMENT_PLAN.md の Next Steps に追記

## 2. 有識者議論プロセス

### 2.1 基本原則

**すべての実装作業の前に、有識者を召喚し多角的に議論してから進めること。**

「とりあえず書いてみる」「動けばいい」は禁止。設計・方針が固まってから手を動かす。

### 2.2 有識者の召喚

作業内容に応じて、以下のような専門家ペルソナを最低3名召喚し、それぞれの視点から議論する。

| ペルソナ | 視点 | 召喚する場面 |
|----------|------|-------------|
| アーキテクト | 全体設計・依存関係・拡張性 | 新機能追加・構成変更 |
| セキュリティ専門家 | 脆弱性・入力検証・依存リスク | 外部連携・API・ユーザー入力 |
| QAエンジニア | テスト容易性・エッジケース・回帰リスク | テスト設計・リファクタリング |
| UXデザイナー | ユーザー体験・操作フロー | UI変更・ワークフロー設計 |
| ビジネスアナリスト | ROI・市場ニーズ・優先順位 | 機能要否の判断・方向性決定 |
| DevOpsエンジニア | CI/CD・デプロイ・環境依存 | インフラ・依存関係変更 |

### 2.3 議論の進め方

1. **課題定義**: 何を実装するか、なぜ必要かを明文化
2. **有識者召喚**: 関連する専門家ペルソナを3名以上指定
3. **多角的議論**: 各有識者の視点から利点・リスク・代替案を洗い出す
4. **スコアリング**: 下記基準で採点（セクション2.4）
5. **判定**: 合計スコアが **8割以上（40点中32点以上）** で実装着手を許可
6. **記録**: 議論結果とスコアを簡潔に記録してから実装開始

### 2.4 スコアリング基準

| # | 評価項目 | 配点 | 観点 |
|---|----------|------|------|
| 1 | 技術的妥当性 | /10 | 選定技術は適切か、過剰・不足はないか |
| 2 | 保守性・可読性 | /10 | 他者（将来の自分含む）が理解・修正できるか |
| 3 | スコープ適合性 | /5 | 現フェーズの目的に合致しているか |
| 4 | テスト容易性 | /5 | ユニットテスト・結合テストが書きやすいか |
| 5 | リスク・副作用 | /5 | 既存機能への影響、障害リスクは許容範囲か |
| 6 | 費用対効果 | /5 | 実装コストに見合う価値があるか |
| | **合計** | **/40** | **32点以上で着手許可** |

### 2.5 スコアリング例

課題: extract_sectionsにsectionタグ・見出し分割フォールバックを追加
有識者: アーキテクト / QAエンジニア / UXデザイナー

1. 技術的妥当性:   9/10 - 正規表現ベースで軽量、Playwrightにも同等ロジック
2. 保守性・可読性: 8/10 - 優先順位が明確、関数分離済み
3. スコープ適合性: 5/5  - 受託対応に直結する必須機能
4. テスト容易性:   5/5  - HTMLパターン別にテスト可能
5. リスク・副作用: 4/5  - 既存data-section優先のため後方互換OK
6. 費用対効果:     4/5  - 実装1時間以内、受託の幅が広がる

合計: 35/40 (87.5%) → 着手許可

### 2.6 8割未満の場合

スコアが8割未満（31点以下）の場合:
- 低スコア項目の改善案を議論する
- 代替アプローチを検討する
- スコープを縮小して再スコアリングする
- それでも8割に届かない場合は実装を見送り、理由を記録する

## 4. ワークフロールール

- push後は必ずリポジトリから直接ファイルを参照する
- 会話圧縮が発生したら RULES.md を再読み込みする
- 次フェーズ移行前に最新コードを取得する
- mainブランチで作業（ソロ開発）、CI green必須
- コミットメッセージ: feat:, fix:, docs:, test:, refactor: プレフィックス

---

## 5. テストルール

- 新規関数にはユニットテストを追加する
- CI (ruff, mypy, pytest) が全てパスすること
- テスト層: 純粋ロジック / 外部依存 (skip-if) / 手動E2E
